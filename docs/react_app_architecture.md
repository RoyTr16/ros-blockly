# React App Architecture

This document describes the architecture of the React-based frontend for the ROS Blockly Robot Control project.

## Overview

The application is designed to provide a clean, modular, and responsive user interface for controlling a ROS robot using Blockly. It leverages modern React patterns, including **Context API** for global state management and **Custom Hooks** for logic encapsulation.

## Core Concepts

### 1. Global State Management (`RosContext`)

We use the React Context API to manage the ROS connection state and logs globally. This avoids "prop drilling" (passing data through many layers of components).

*   **`RosContext.jsx`**:
    *   Initializes the `roslib` connection using a 3-tier URL fallback:
        1.  `localStorage.getItem('rosbridge_url')` (user override)
        2.  `import.meta.env.VITE_ROSBRIDGE_URL` (env var from Docker/Vite)
        3.  `ws://${window.location.hostname}:9090` (auto-detect)
    *   Manages connection status (`connected`).
    *   Maintains a list of timestamped logs (`logs`).
    *   Provides an `addLog` function to components.
    *   Initializes `window.rosBlockly` for global runtime state (intervals, graph data).
    *   **Usage**: Any component can access these values using the `useRos()` hook.

### 2. Logic Encapsulation (`useRobotControl`)

Complex logic for controlling the robot is abstracted into a custom hook, keeping UI components focused on rendering.

*   **`useRobotControl.js`**:
    *   Consumes `RosContext` to get the ROS connection.
    *   **`runCode()`**: Executes the JavaScript code generated by Blockly. It wraps code in an async IIFE using `new Function()` with access to `ros`, `ROSLIB`, `log`, and `wait`. Supports **concurrent execution** — when the workspace contains multiple disconnected block groups, they run in parallel via `Promise.all`.
    *   **`stopExecution()`**: Cancels a running program using an `AbortController`. The `wait` function checks the abort signal, allowing clean mid-execution stops.
    *   **`resetRobot()`**: Stops any running execution, clears intervals/subscriptions/graph data, then iterates over **package-defined reset actions** from `PackageLoader.getAllResetActions()` — publishing stop commands and calling services as defined by each loaded package.

### 3. Component Hierarchy

*   **`App.jsx`**: The root component. Wraps the application in `RosProvider`. Defines the main layout: `Header` → `BlocklyComponent` (with `GraphOverlay`) → toggleable `ControlPanel`.
*   **`Header`**: Displays the title, connection status indicator, and all action buttons (Run, Stop, Reset, Save, Load). Also contains the toggle button for the side panel.
*   **`BlocklyComponent`**: Wraps the Google Blockly workspace. Handles workspace injection and reports generated code, code groups, and preamble back to `App` via `onCodeChange`.
*   **`GraphOverlay` (GraphViewer)**: A transparent overlay on top of the Blockly workspace that renders real-time graphs (e.g., sensor data plots) driven by `window.rosBlockly.graphs`.
*   **`ControlPanel`**: A toggleable right-hand sidebar with two tabs:
    *   **AI Chat tab**: Contains the `AiChat` component for natural-language interaction with the workspace.
    *   **Console tab**: Contains `CodeDisplay` (generated JavaScript), `FunctionPanel` (reusable function library), and `LogViewer` (system logs).

## Styling

*   **CSS Files**: We use standard CSS files imported by components (one `.css` per component).
*   **Flexbox**: The layout relies heavily on Flexbox for a responsive design that fills the screen height (`100vh`).
*   **Theme**: A dark/light theme mix with a professional look (dark header, clean white panels).

## Key Workflows

### Connecting to ROS
On mount, `RosContext` resolves the WebSocket URL (localStorage → env var → auto-detect) and connects. Event listeners update the `connected` state, which updates the UI automatically.

### Running Code
1.  User drags blocks in `BlocklyComponent`.
2.  `BlocklyComponent` generates JavaScript via the package-defined generators (loaded by `PackageLoader`) and reports the code, code groups, and preamble to `App.jsx` via `handleCodeChange`.
3.  User clicks "Run" in the `Header`.
4.  `useRobotControl.runCode()` wraps the code in an async IIFE and executes it via `new Function()`.
    *   If there are multiple disconnected block groups, they run concurrently with `Promise.all`.
    *   Single block groups run sequentially.
5.  The code uses `roslib` to publish/subscribe to ROS topics.

### Stopping Execution
1.  User clicks "Stop" in the `Header`.
2.  `useRobotControl.stopExecution()` calls `abort()` on the `AbortController`.
3.  Any pending `await wait(...)` calls reject immediately, halting the program.

### Resetting Robot
1.  User clicks "Reset" in the `Header`.
2.  `useRobotControl.resetRobot()` aborts any running execution and clears runtime state (intervals, ultrasonic subscriptions, graph data).
3.  It calls `getAllResetActions()` from `PackageLoader` and executes each action — publishing to topics and/or calling services as defined by the loaded robot packages.
