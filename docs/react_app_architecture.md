# React App Architecture

This document describes the architecture of the React-based frontend for the ROS Blockly Robot Control project.

## Overview

The application is designed to provide a clean, modular, and responsive user interface for controlling a ROS robot using Blockly. It leverages modern React patterns, including **Context API** for global state management and **Custom Hooks** for logic encapsulation.

## Directory Structure

The `client/src` directory is organized as follows:

```
client/src/
├── components/
│   ├── blockly/          # Blockly-related components
│   │   └── BlocklyComponent.jsx
│   └── ui/               # Reusable UI components
│       ├── ActionButtons.jsx
│       ├── CodeDisplay.jsx
│       ├── ControlPanel.jsx
│       ├── Header.jsx
│       └── LogViewer.jsx
├── context/              # Global state providers
│   └── RosContext.jsx
├── hooks/                # Custom logic hooks
│   └── useRobotControl.js
├── generators/           # Blockly code generators
│   └── ros_generator.js
├── App.jsx               # Main application entry point
└── App.css               # Global styles
```

## Core Concepts

### 1. Global State Management (`RosContext`)

We use the React Context API to manage the ROS connection state and logs globally. This avoids "prop drilling" (passing data through many layers of components).

*   **`RosContext.jsx`**:
    *   Initializes the `roslib` connection to the WebSocket server (`ws://localhost:9090`).
    *   Manages connection status (`connected`).
    *   Maintains a list of logs (`logs`).
    *   Provides an `addLog` function to components.
    *   **Usage**: Any component can access these values using the `useRos()` hook.

### 2. Logic Encapsulation (`useRobotControl`)

Complex logic for controlling the robot is abstracted into a custom hook, keeping UI components dumb and focused on rendering.

*   **`useRobotControl.js`**:
    *   Consumes `RosContext` to get the ROS connection.
    *   **`runCode()`**: Executes the JavaScript code generated by Blockly. It creates a sandboxed function with access to `ros`, `ROSLIB`, and `log`.
    *   **`resetRobot()`**: Stops the robot and calls the `/world/empty/set_pose` service to reset its position in Gazebo.

### 3. Component Hierarchy

*   **`App.jsx`**: The root component. It wraps the application in `RosProvider` and defines the main layout (Header + Main Content).
*   **`Header`**: Displays the title and connection status (Green/Red indicator).
*   **`BlocklyComponent`**: Wraps the Google Blockly workspace. It handles the injection of the workspace and updates the generated code on change.
*   **`ControlPanel`**: The right-hand sidebar containing:
    *   **`CodeDisplay`**: Shows the generated JavaScript code.
    *   **`ActionButtons`**: "Run Code" and "Reset Robot" buttons.
    *   **`LogViewer`**: A scrollable list of system logs.

## Styling

*   **CSS Modules**: We use standard CSS files imported by components.
*   **Flexbox**: The layout relies heavily on Flexbox for a responsive design that fills the screen height (`100vh`).
*   **Theme**: A dark/light theme mix with a professional look (dark header, clean white panels).

## Key Workflows

### Connecting to ROS
On mount, `RosContext` attempts to connect to `ws://localhost:9090`. Event listeners update the `connected` state, which updates the UI automatically.

### Running Code
1.  User drags blocks in `BlocklyComponent`.
2.  `ros_generator.js` converts blocks to JavaScript.
3.  `App.jsx` receives the code via `handleCodeChange`.
4.  User clicks "Run Code".
5.  `useRobotControl` executes the code string.
6.  The code uses `roslib` to publish to `/cmd_vel`.

### Resetting Robot
1.  User clicks "Reset Robot".
2.  `useRobotControl` publishes a stop command (0 velocity).
3.  It then calls the Gazebo service `/world/empty/set_pose` to teleport the robot back to the origin.
